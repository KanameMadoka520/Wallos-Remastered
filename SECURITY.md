# Wallos-Remastered 安全策略

## 安全目标

当前分支重点保护：

- 账号安全
- 私有图片资源
- 备份文件
- 管理员高危操作
- 上传处理链路

## 已落实的措施

### 账号与认证

- 登录失败限速
- 管理员临时密码重置但不提供明文密码查看
- CSRF 页面安全令牌服务端强制 30 分钟过期
- 页脚只显示 CSRF 短指纹，不泄露完整令牌
- 登录会话保留时间和 CSRF 令牌时间是两套机制；勾选长期登录时，会话仍可保留更久，但表单安全令牌仍需定期刷新
- 受保护异步接口未登录时统一返回 `401 session_expired` JSON，CSRF 失效时统一返回 `400 invalid_csrf` JSON
- 前端共享请求层会把意外返回的登录页 HTML 归一化为会话失效，避免旧脚本或代理漂移导致用户只看到未知错误

### 订阅图片

- 订阅图片静态直链被拦截
- 媒体访问必须鉴权
- 普通用户只能访问自己的图片
- 管理员可以跨用户审核图片

### 文件上传

- 上传格式限制
- 服务端校验图片尺寸、MIME、扩展名和 GD 解码能力
- 未压缩订阅原图会透传保存，但预览图/缩略图仍会重新编码生成，且订阅图片目录禁止静态直链访问
- 尺寸、大小与内存预算限制

### 备份与恢复

- `/backups/` 静态目录被拦截
- 备份下载走后台鉴权端点
- 恢复前先做校验
- 新备份包含校验清单

### 缓存与运维

- 管理员后台可查看当前 Service Worker 控制状态
- 管理员可以清理当前浏览器缓存，并发布客户端静态资源刷新提示
- Service Worker 不缓存订阅私有媒体目录 `images/uploads/logos/subscription-media/`，避免已删除图片或其他账号图片在浏览器共享缓存里残留
- Service Worker 和版本化 CSS/JS 会随文件版本主动刷新；客户端缓存刷新提示必须持续显示到用户手动关闭，避免用户错过静态资源更新提醒
- 管理员维护入口可查看日志保留策略、存储占用、日志增长风险、订阅图片孤儿文件和缺失派生图记录
- 订阅图片审计 CSV 只导出扫描结果，不会自动删除文件
- 管理员可手动清理数据库不再引用的订阅图片孤儿文件；清理范围被限制在订阅图片目录内，避免误删其他持久化文件
- SQLite `PRAGMA optimize` / `ANALYZE` / `VACUUM` 只能由管理员手动触发，维护结果会显示前后数据库大小和空闲页变化；`VACUUM` 可能短暂锁定写入

## 运行建议

- 始终持久化 `db`、`logos`、`backups`
- 定期抽测备份校验与恢复
- 不要将调试入口直接暴露到公网
- 如果页面打开超过 30 分钟，提交表单前先刷新页面，以避免 CSRF 令牌过期
- 定期查看访问日志、安全异常、速率限制记录和订阅图片审计结果
- 图片、日志、支付记录持续增长时，先查看后台存储占用与日志增长风险，再定期安排 SQLite 维护窗口

## 不接受的危险实现

- 明文密码查看
- 私有图片或备份的公开直链
- 跳过上传校验或允许订阅图片目录被公网直接访问
- 不校验直接恢复备份

### 2026-04 额外加固
- “禁用登录”旁路只允许本地直连请求触发，不再适用于通过公网域名、FRP、反向代理或转发头访问的请求。
- 空实例的首次数据库恢复已限制为本地直连访问，并统一复用现有备份校验与恢复链路。
- API key 不再建议通过 URL 查询参数传递，推荐使用 `X-API-Key` 或 `Authorization: Bearer ...` 请求头。
- 支付方式图标的外链抓取已补充逐跳重定向校验与地址约束，用于降低 SSRF 风险。
- 认证相关 cookie 会在 HTTPS 或可信 HTTPS 转发场景下自动补充 `Secure` 标记。
- CSRF 页面安全令牌现在服务端强制 30 分钟 TTL；过期后继续提交会返回 `invalid_csrf`，用户应刷新页面重新加载安全令牌。


## 上游安全补丁同步策略

- 本项目不直接合并官方 Wallos 页面结构，而是按补丁含义手动移植安全修复，避免覆盖 Remastered 的主题、交互和运维增强。
- 已手动吸收上游 Wallos 4.8.1 / 4.8.3 的关键安全含义：SSRF / DNS rebinding 防护、CGNAT 检测、非管理员内网目标限制，以及私有页面输出转义。
- 远程 logo、支付图标、AI 本地模型端点和通知 webhook 这类服务端外联请求必须经过地址校验；标准用户不能把目标指向内网或保留地址。
- 管理员如确需访问内部 webhook 目标，应使用后台 allowlist 明确放行，并避免把不可信用户提升为管理员。

## 管理员后台测试凭据

- 系统仍坚持只有 id = 1 是管理员，不支持额外管理员账号，避免扩大高权限攻击面。
- 管理员后台浏览器 E2E 只接受显式提供的管理员用户名/密码或短期 Cookie。
- 运行管理员 E2E 时必须同时提供 WALLOS_ADMIN_AUTH_EXPIRES_AT，脚本会在凭据缺失、过期或时间格式无效时安全跳过。
- 不要把管理员密码、管理员 Cookie 或未过期的测试凭据写入仓库、CI 配置、截图或长期脚本。
- 订阅图片 E2E 使用普通测试账号，真实上传临时 PNG 并验证原图透传、派生图大小上限、未登录拒绝访问和删除后不可读；测试结束会清理临时订阅。
