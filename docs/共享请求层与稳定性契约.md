# 共享请求层与稳定性契约

## 目的

这份文档用于约束未来开发，防止仓库再次退化成：

- 高频页面各自手写 raw `fetch`
- 会话失效处理到处散落
- 错误提示每页一套
- 改完高风险链路却不做最小回归

## 1. 高频页面默认规则

以下类型的请求，默认应优先复用共享请求层：

- 管理员页保存/更新操作
- 设置页的高频 CRUD 与保存操作
- 日历、订阅等主页面的常规 JSON / Form 请求

推荐优先顺序：

1. `WallosApi`
2. `WallosHttp`
3. 在局部脚本中基于以上两者做小型 helper

不推荐继续新增：

```js
fetch(url, ...)
  .then(response => response.json())
  .then(...)
  .catch(...)
```

尤其是不应在每个高频页面里重复手写相似的：

- `response.ok` 判断
- `session_expired` 判断
- 通用错误提示
- 成功提示

## 2. 哪些流可以暂时保留原始实现

以下场景允许继续保留原始实现，前提是它们确实属于特殊流：

- 下载/导出
- 纯文本 cronjob 输出
- 纯二进制媒体流
- 少量一次性、极低频的特殊调用

这类保留不代表“永远不收敛”，只是说明它们不应和高频 JSON / Form 请求混用同一规则。

## 3. 会话失效契约

高风险受保护异步接口目前应尽量遵守统一的 session 失效契约，至少包括：

- `success: false`
- `code: "session_expired"`
- `error: "session_expired"`
- `message`

前端侧应优先通过共享请求层识别这一契约，而不是在业务模块重复硬编码：

- `status === 401`
- 字符串比较 `session_expired`
- 各自 reload / redirect

## 4. 共享反馈层

关键失败提示优先复用：

- `common.js` 里的 toast
- 请求层统一错误归一化
- 共享 session-expired 处理入口

目标是：

- 用户看到更明确的失败反馈
- 维护者减少每页重复处理逻辑

## 5. 回归基线

高风险链路改动后，建议至少跑一次：

```bash
docker exec wallos-local php /var/www/html/tests/regression_runner.php --base-url=http://127.0.0.1
```

当前 regression runner 已经覆盖：

- `health.php`
- 登录页 / 注册页基础契约
- `theme-color`
- Service Worker 缓存常量
- 关键受保护 endpoint 的 unauthenticated 401 契约
- 旧 PHP 逻辑回归测试

## 6. 未来开发者最容易犯的退化

下面这些做法最容易让仓库重新变乱：

- 在高频页面继续新增 raw `fetch`
- 在业务模块本地重复写错误/成功处理
- 新接口绕过共享请求层
- 改完高风险链路不跑 regression
- 文档不跟随架构收敛同步更新
